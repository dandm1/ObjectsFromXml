ObjectsFromXML

Developer Guide

Synopsis

The ObjectsFromXml library is a simple C# library to convert a structured XML document into an array of .NET objects for use wherever an array of such objects is required in .NET code. The routine has the following features:

        Can construct an array of any type of objects that can be described by initialising properties with POCOs or other objects that can, in turn, be described in this way.

        Uses reflection to inspect types and populate type properties from either XML attributes or nodes.

        Provides for common Properties which will be applied to all objects with a given property name.

        Allows objects created earlier in the file to be referenced as parameters to other objects later in the file.

        Provides for resources which can be used in any compatible context throughout the file.

        Extensive type checking and logging for easy diagnosis.

Main Classes / Methods

The main class for the object builder is ObjectBuilder. The key external methods to this are:

-        Constructor takes two inputs, a text stream and an output object type.

-        Logger property allows a class that implements ILog to be assigned to the builder. If an appropriate class is assigned, this will be used to output logging information on the conversion process.

-        Build method builds the objects described in the input stream, returning an IEnumerable of the specified type. This exists in either a basic form which returns the results as an IEnumerable of Object types or in a generic Build<Type> version that returns a strongly typed IEnumerable.

Other important classes in the library are:

-        ILog interface when implemented allows log information from the building process to be accessed. The structure of this mimics log4net for easy wrapping to ensure log information becomes part of the overall program log.

-        ObjectBuilderAttribute a class attribute, used to identify classes which can be included in the objects created.

-        IExtraParameters interface when a class that can be built implements this interface, any values passed to it that do not match properties will instead be passed to a Dictionary<string,dynamic> named Parameters.

Main Program Flow

When the ObjectsFromXml library is first created, it uses reflection to identify any DLLs in the current operating directory which contain classes which it can instantiate. To achieve this, it looks for any DLLs in the directory that contain classes decorated with the ObjectBuilderAttribute. When such a DLL is found, the DLL is copied to the system Temp directory and loaded from there. Each time Build is called the DLLs are re-checked and any new or changed files will be copied for use. This permits the targeted DLLs to be updated without the need to restart the target application.

The main call proceeds from the start of the input stream, constructing first any common Properties, then any resources and finally the target objects. A list of basic POCO types (see appendix 1) are available automatically in the system as well as any classes found in copied DLLs. Types must be decorated with the ObjectBuilderAttribute wherever they appear in the XML description (that is whether they are the final output object or simply used to set properties of those types). To be usable in the object builder, types must contain a constructor which takes either no parameters or a single string parameter. Any other information needed for the type must be set through properties.

XML Format

The XML provided to the builder can be any type of text stream though will usually come from a text file and so it will be referred to as a file in this documenation.

The root node of the input stream must be of type ObjectSet. The nodes acceptable below this are:

-        Common for common Properties to be passed to all objects.

-        Resources for named resources that can be used anywhere in the file

-        Objects for the target objects to be returned from the Build() call.

Object Format

In any of these sections, a common structure is used to describe the objects to be built.

Each object is built from a base node the name of which is the type to be built. This node can then have attributes and contain child nodes with names that match the names of properties on the target class.

Attributes can be used to set string properties and properties of any other types which can be constructed using a static object.Parse(string) structure. Child nodes can be used to set properties of any type which is available to the object builder with any children of the Property Node being constructed as full objects in their own right (and so able to contain other properties which are set using other child objects, and so on).

Where the Property to be set is an array type (IEnumerable, List<> or [] Array) the node may simply contain multiple entries which will be converted into an appropriate array type for passing to the Property. Where the Property to be set is a Dictionary<,> each node in the property must be named Entry and contain two child nodes or attributes named Key and Value.

Examples

A simple string containing the text Hello World!:

<string>Hello World!</string>

A FileInfo object reflecting C:\temp.txt :

<FileInfo>C:\temp.txt</FileInfo>

A datetime object initialised to 20 June 2016 where the text will be passed to the DateTime.Parse method to create the output. ISO date format is shown, and recommended, though any format parsable using the location settings for the running PC will work.

<Datetime>2016-06-20</Datetime>

A FileOpener object constructed using a constructor with no parameters and then with the Name, Filename and FileMode properties set.

<FileOpener Name="Open File" Filename="this.txt" FileMode="Read"/>

A TextReader object with its name set to Read Text and its format set to HTML. The Property Numbers is an array of doubles and will be initialised with a four element array.

<TextReader Name="Read Text">

<Format>

<string>HTML</string>

</Format>

<Numbers>

<double>1.0</double>

<double>3.14159265</double>

<double>1000000000</double>

<double>-2.17159</double>

</Numbers>

</TextReader>

A custom Converter object containing a Mapping property with a list mapping strings onto dates. The Mapping property here is of type Dictionary<string,DateTime>. The XML structure demonstrates that it is possible to mix approaches to defining Keys and Values if that is appropriate:

<Converter>

<Mapping>

<Entry Key=Australia Day Value=2016-01-26/>

<Entry Key=Good Friday Value=2016-03-29/>

<Entry>

<Key>Easter Monday</Key>

<Value>2016-04-01</Value>

</Entry>

<Entry Key=Christmas Day><Value>2016-12-25</Value></Entry>

</Mapping>

</Converter>

<Ref /> Nodes

In any context where an object can be explicitly defined, a Ref node can be used to include a reference to a previously constructed object in its place. A Ref node takes two attributes:

-        Source which is the Name of a previously defined object, and

-        Optionally Property which is the name of the property on that object to get the data from.

The Source given for must exactly match the Name attribute on a node defined earlier in the file. Where a Ref object with a valid Source is found as a parameter, that parameter will be assigned to a reference to the previously constructed object. If multiple target objects reference the same source object a shared copy of the object will be passed to all of them allowing each step to make changes to it.

For example, the following code defines three simple objects:

       a FileOpener which opens a text file,

       a FileReader which loads the file and has the previously constructed FileOpener passed to it as a prerequisite, and

       a TextBoxDisplay which displays the file length by taking a property named Length from the FileReader and loading it into its Value property.

<ObjectSet>

<FileOpener Name=FileOpen Filename=temp.txt />

<FileReader Name=FileRead >

<Prerequisites><Ref Source=FileOpen/></Prerequisites>

</FileReader>

<TextBoxDisplay>

<Value>

<Ref Source=FileRead Property=Length/>

</Value>

<Prerequisites><Ref Source=FileRead/></Prerequisites>

</TextBoxDisplay>

</ObjectSet>

When used to populate a multi-entry property (a property that implements IEnumerable) a reference can also be to a property that itself returns a list.

List, Array, Enumerable and Dictionary Nodes

When it is appropriate for the target data types, it is possible to explicitly create lists by using a one of the first three of these types. As noted earlier, in most contexts it is not necessary to explicitly specify that a list is to be created however it may be required when the builder is unable to infer from context that a list is to be created (for example in Common and Resource nodes).

A dictionary node is more complicated, as it requires two entries per item. Each child node of a dictionary (either explicit or implied) should be of type Entry and must have a record named key (which can be either a child node or an attribute). It would typically also have a record named Value (again, either a node or an attribute) and if this is omitted a value of Null will be used.

On an explicit Dictionary tag, two attributes should be specified KeyType and ValueType. These specify (respectively) the types for the keys and values in the Dictionary to be constructed and must resolve to types that exist either in the standard C# types in Appendix 1 or in a custom DLL.

The <Common> node

One of the three supported nodes below the ObjectSet node at the top level of an object XML file is a <Common> node. This node allows the definition of property values that will be passed to every object in the file which contains a Property with a matching name and a compatible type.

As the objects constructed in this section will be applied to all properties with a given name they begin with a node which specifies the name of the property to which they will be applied which is followed by a definition of the object to be constructed.

This is useful for creating objects which can be shared between all the items in a file and used for communication as well as for defining common context parameters. The <Common> node is optional.

For example, the following XML creates an instance of a class named CommunicationVariables which will be passed to every object in the file which has a compatible property named Variables without this needing to be explicitly specified.

<ObjectSet>

<Common>

<Variables>

<CommunicationVariables Name=CommVars Context=FooBar />

</Variables>

</Common>

<Objects>

<Job1/>

<Job2/>

</Objects>

</ObjectSet>

The <Resources> node

The resources node is used to create named objects of any type which can be referred to by name later in the file. Any objects specified in this section will be created and stored in the main list of named objects. They will not however be used for anything until they are referenced later in the file.

As the objects to be constructed in this section of the file are not directly used for properties each object definition starts with the type to be constructed directly.

This structure is useful for creating complicated objects which are needed repeatedly as well as for creating objects which need to be passed to some, but not all, of the output objects.

For example, the following code creates a Colour object named LimeGreen which is used as a text colour for one of the subsequent display objects and a background for another one:

<ObjectSet>

<Resources>

<Colour Name=LimeGreen>

<Red>219</Red><Green>226</Green><Blue>0</Blue>

</Colour>

</Resources>

<Objects>

<Heading Text=This is the Heading>

<BackgroundColour><Ref Source=LimeGreen/></BackgroundColour>

</Heading>

<Content>

<Text><p>This is body text.</p><p>It is not in a nice colour.</p></Text>

<TextColour><Ref Source=LimeGreen/></TextColour>

</Text>

</Objects>

</ObjectSet>

The <Objects> Node

The objects node is the final node type which is valid inside the root <ObjectSet> node. This node contains the details of the objects to be created and included in the function output. This node is required in the file.

Each object in this section has a base node consisting of the type to be constructed. There are no limit to the number of nodes that can be included in this list.
